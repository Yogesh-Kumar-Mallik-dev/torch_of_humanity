# Contributing to Torch of Humanity

Thank you for contributing to Torch of Humanity.

This is a founder-led, source-available game project. All contributions must follow this document, the repository standards, and the THFO-1.1 license.

## Project Nature

- This project is not a conventional open-source project.
- The code is source-available for learning, collaboration, and approved development.
- The game is intended for eventual public/commercial release.
- Founder and Core Team governance applies within the scope of this project.

## Ownership and License

By contributing, you agree to the four-layer ownership model defined in LICENSE:

- Layer 1: Founder
- Layer 2: Core Team
- Layer 3: Contributors
- Layer 4: Community Users

Contributor code ownership is retained by the contributor, while merged contributions are licensed for project use as defined in THFO-1.1.

## Repository Structure

Contributions should align with the current project layout:

```text
.
├── main.lua
├── config.lua
├── core/
├── engine/
├── lib/
└── docs/
```

## Contribution Workflow

1. Create a focused branch.
2. Implement one logical change set per pull request.
3. Keep changes modular and consistent with existing code style.
4. Update relevant docs in docs/ when behavior changes.
5. Submit a PR with clear scope, rationale, and testing notes.

## Coding Expectations

- Keep modules small and purpose-driven.
- Prefer reusable systems over one-off patches.
- Avoid unnecessary globals.
- Maintain readability and maintainability.
- Preserve performance-conscious design choices.

## Commit Policy (Mandatory)

This project enforces strict commit formatting.

- Always use the repository .gitmessage template.
- Refer to previous commits and match their structure exactly.
- Keep commit messages consistent in style, scope, and format.
- Pull requests that do not follow this standard will be automatically rejected.

## Pull Request Requirements

Each PR must include:

- A concise summary of what changed.
- Why the change is needed.
- Any validation or manual test notes.
- Screenshots or clips for visible gameplay/UI changes, when applicable.

## Rejection Conditions

A contribution may be rejected if it:

- Breaks existing behavior or architecture.
- Conflicts with project direction.
- Ignores repository or commit standards.
- Introduces unnecessary complexity.
- Omits required documentation updates.

Merge is at the discretion of the Founder and Core Team.

## Credits and Release Attribution

For public releases, credit/reference obligations follow the license terms.

- Founder, Core Team, and Contributors are entitled to credit/reference.
- Credits must appear in at least one official release channel.
- If a member of the top three layers does not want their name shown, they must submit an opt-out through email or another written binding document.

## Communication Standards

- Keep discussions respectful and technical.
- Be precise and constructive in reviews and issue threads.
- Focus feedback on project quality and long-term maintainability.

## Final Authority

Within the Torch of Humanity project scope, final governance authority rests with:

- Founder
- Core Team
